Аннотация

Этот проект переводит исходный корпус данных, первоначально организованный в Kumu-совместимых CSV, в воспроизводимый Python-пайплайн. Итоговый результат строится не в Kumu, а собственным кодом: данные загружаются, нормализуются, частично дополняются автоматизированным сбором с официальных сайтов, после чего на их основе строятся графы, считаются сетевые метрики и создаются визуализации.

Исследовательский вопрос

Насколько формальная структура российской власти совпадает с сетевой структурой кадровых, координационных и аналитически реконструируемых связей?

Задачи

  1. Собрать и объединить формальный и аналитический слои данных о российской власти.
  2. Построить формальный институциональный граф и сравнить его с дополнительными слоями сети.
  3. Вычислить базовые сетевые метрики центральности для formal core.
  4. Визуализировать различия между слоями и показать ограничения такого моделирования.

Данные

Проект использует четыре слоя данных:

  • official — формальный институциональный слой, собираемый кодом с официальных сайтов;
  • core — формальный институциональный каркас исследовательской базы;
  • overlay — аналитический слой на основе политологических исследований;
  • comention — слой соупоминаний, фиксирующий дополнительную связанность акторов.

Изначальные Kumu-таблицы используются только как сырой формат хранения части данных. Финальный анализ, очистка, объединение, расчёт метрик и визуализация выполняются Python-кодом.

Автоматизированный сбор данных

В проект включён отдельный модуль src/scrape_official.py, который пытается автоматически собрать часть формального институционального слоя с официальных сайтов. На текущем этапе он ориентирован на страницы:

  • Кремля;
  • Правительства РФ;
  • Государственной Думы;
  • Совета Федерации.

Для воспроизводимости в офлайн-среде в репозитории хранится кэшированный snapshot официального слоя. Это означает, что даже без доступа к сети можно воспроизвести структуру проекта и показать отдельный этап автоматизированного сбора данных.

Методика

Использованные цифровые методы

  1. Автоматизированный сбор и структурирование официальных данных с официальных сайтов государственных институтов.
  2. Сетевой анализ формального институционального графа (networkx): degree centrality, betweenness centrality, closeness centrality.
  3. Сравнительный анализ многослойной сети: сопоставление official, core, overlay и comention по размеру, типам связей и плотности.

Технический пайплайн

  1. Загрузка сырых CSV из data/raw/.
  2. Нормализация и объединение узлов и связей в data/processed/.
  3. Построение formal core графа по основным формальным связям.
  4. Расчёт центральностей и генерация таблиц результатов.
  5. Построение статических визуализаций Python-кодом.

Подграф formal core

Этот граф показывает наиболее связанные элементы формального ядра. Он полезен как опорная карта институционального устройства, но не должен автоматически отождествляться с «реальным распределением власти»: для этого и нужен аналитический слой.

Интерпретация

Основные выводы

  • Формальный слой core существенно менее плотный, чем слой comention, что ожидаемо: официальная институциональная структура задаёт более жёсткий и ограниченный набор связей.
  • Наиболее центральным узлом formal core выступает институт президента, а среди связанных институциональных узлов заметны Правительство РФ, Госдума и Совет Федерации.
  • Сравнение слоёв показывает, что формальная архитектура власти и аналитически реконструированная сеть не совпадают полностью: разные слои фиксируют разные механизмы связанности.
  • Даже ограниченный автоматизированный сбор официального слоя улучшает воспроизводимость проекта и делает институциональную часть менее зависимой от ручного ввода.

Ограничения

  • Автоматизированный сбор официальных данных пока охватывает только часть институционального слоя.
  • Слои overlay и comention не являются полностью verified-only и должны интерпретироваться как аналитическая реконструкция.
  • Значения центральности зависят от того, какие типы рёбер включены в formal core.
  • Первоначальная структура части данных была совместима с Kumu, но финальный анализ выполнен собственным Python-кодом.

Воспроизводимость

Структура репозитория

  • src/ — код загрузки, построения графов, расчёта метрик, визуализации и автосбора;
  • scripts/ — сценарий полного запуска;
  • data/raw/ — исходные CSV;
  • data/processed/ — очищенные таблицы;
  • outputs/ — таблицы результатов и готовые визуализации;
  • report/ — Quarto-отчёт.

Запуск

pip install -r requirements.txt
python scripts/run_analysis.py
quarto render report/index.qmd

Заключение

Проект показывает, что корпус данных, изначально организованный для Kumu, можно превратить в воспроизводимый Python-пайплайн, который включает автоматизированный сбор официальных данных, сетевой анализ, сравнение слоёв сети и кодовые визуализации. Это делает результат более академически прозрачным и лучше соответствует требованиям итогового проекта.
